The AI Resume Assistant can use a curated set of tools while it chats with you. You do not call these tools directly; the assistant chooses them when your request needs resume data, supported provider web context, a follow-up question, or a resume patch.

## Tool activity in chat

Tool activity appears inside the conversation. Some activity is collapsed by default so the chat stays readable.

Applied resume patches are shown as a **Resume updated** marker in chat. The builder preview refreshes after the patch is applied.

## Available tools

| Tool | What it does | Example request |
| --- | --- | --- |
| `read_resume` | Reads the active resume and gives the assistant the resume data it can safely edit. | "What are the weakest parts of this resume?" |
| `web_search` | Uses the selected provider's native web search when that provider/model supports it. | "Research this company and adjust the summary for its product area." |
| `ask_user_question` | Shows a question card with answer choices when the assistant needs your decision. | "Ask me before changing the career narrative." |
| `apply_resume_patch` | Applies a JSON Patch to the active resume data. | "Change the visible name to Amruth Pillai." |

## Resume patches

Resume patches are rooted at the resume data object. For example, the visible resume name is patched at `/basics/name`.

When a patch is applied:

- the active resume updates immediately;
- the chat shows a **Resume updated** marker;
- the builder preview refreshes to show the updated resume;
- the shared builder undo stack captures the change.

Use the builder dock's undo and redo buttons to roll back or restore assistant-applied patches. If the resume changed after the assistant generated a patch, applying it can fail with a version conflict. In that case, ask the assistant to retry from the latest resume.

## Web access

Live web research is handled only by the selected AI provider's native web search tool. When the provider/model supports it, the assistant can use `web_search` for current company, industry, role, or URL-based context.

When the provider/model does not support native web search, the assistant still works for normal resume editing. If you ask it to browse, search the web, fetch a URL, or use current online context, it should tell you that live web research is unavailable with the selected provider/model and ask you to paste the relevant content instead.

For self-hosted deployments:

- app-owned URL crawling is not available;
- web access depends on the selected provider/model supporting native web search;
- unsafe/private AI provider base URLs require `FLAG_ALLOW_UNSAFE_AI_BASE_URL=true`, which should only be used on trusted self-hosted deployments.

## Good prompts for tool use

Use direct prompts that tell the assistant what context to use and how cautious to be:

- "Research this role, identify the most important keywords, and apply a conservative patch."
- "Use the pasted job description and ask me before changing anything outside the summary."
- "Compare my current projects against this company page and suggest only truthful wording."
- "Apply a patch for the visible resume name, then show me what changed."

## When tools are unavailable

Tool use can be limited by the selected provider, deployment configuration, or resume state.

- A deleted, disabled, or untested provider blocks new assistant runs until a provider is enabled and tested again.
- A deleted or locked resume blocks editing.
- Live web research is unavailable when the selected provider/model does not support native web search.
